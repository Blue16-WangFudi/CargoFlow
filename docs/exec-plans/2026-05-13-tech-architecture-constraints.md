# CargoFlow 技术选型与架构约束

## 任务来源

- 任务：技术选型与架构约束
- 来源：PRD 9 / 一期端到端交付：工程基础
- 验收标准：明确前端、后端、数据库、MQTT、地图、RAG、部署方式和本地启动命令，并写入 README 或执行计划。

## 选型结论

| 领域 | 选型 | 约束 |
| --- | --- | --- |
| 前端 | React + TypeScript + Vite | 按角色拆工作台路由；UI 实现前必须阅读 `DESIGN.md`；接口类型从 OpenAPI 或共享契约生成。 |
| 后端 | Python FastAPI | 所有业务能力通过 HTTP API 暴露；接口层必须执行角色权限守卫；OpenAPI 是前后端契约源。 |
| 数据库 | PostgreSQL | 保存核心业务对象、状态流转、告警、通知、审计和问答记录；禁止把业务状态只存在缓存或消息队列。 |
| 迁移 | Alembic | 每个持久化模型变化必须有可回滚迁移；迁移随 PR 一起进入质量门禁。 |
| 缓存与异步 | Redis + Celery | ETA 计算、告警检测、通知发送、RAG 索引等非请求主路径工作放入异步任务。 |
| MQTT | EMQX 本地开发，生产使用托管 MQTT 或自管 EMQX | 设备上报和指令下发必须经过 topic、payload、幂等键和时间戳校验。 |
| 地图 | 地图提供商适配器 | 业务代码只依赖 `MapProvider` 接口；本地默认使用离线几何实现；生产适配 AMap、Baidu 或交付环境确认的供应商。 |
| RAG | pgvector + 检索服务接口 | 一期知识库先覆盖物流规则和 FAQ；业务数据检索必须先做用户授权过滤。 |
| 部署 | Docker Compose 基线 | 本地、CI smoke 和预演环境保持相同容器边界：web、api、worker、db、redis、mqtt。 |

## 模块边界

### Web 前端

- `apps/web` 承载角色入口、货主详情、调度地图、告警处理、司机上报、管理员日志和智能问答页面。
- 前端不得硬编码角色权限结果；所有可见数据仍以后端授权结果为准。
- 地图组件通过后端返回的标准位置、轨迹和告警节点渲染，不直接访问数据库或 MQTT。

### API 服务

- `apps/api` 承载 REST API、认证上下文、权限策略、领域服务和集成适配器。
- API 返回应包含业务状态和不可计算原因，例如位置延迟、ETA 不可计算、无轨迹数据。
- 每个写接口必须记录操作人、时间和原因字段，支撑 PRD 的追溯要求。

### 后台任务

- 后台任务负责 MQTT 消息落库、ETA 刷新、异常检测、通知发送和 RAG 索引。
- 异步任务必须可重试且幂等；以设备消息 ID、指令 ID 或业务事件 ID 作为幂等键。

## 核心数据约束

- 货物、车辆、运输任务、位置点、告警、调度指令、状态上报、通知、审计日志、问答记录是第一阶段核心模型。
- 运输任务状态按 PRD 定义流转：`待绑定 -> 已绑定 -> 已装货 -> 运输中 -> 已送达 -> 已签收`，`已取消` 为终态分支。
- 告警状态按 PRD 定义流转：`待处理 -> 处理中 -> 已关闭`，`误报` 为人工确认终态。
- 调度指令状态按 PRD 定义流转：`待下发 -> 已下发 -> 已送达 -> 已确认`，失败和撤销为终态分支。
- GPS 上报必须校验经纬度范围、采集时间、设备编号、任务关联和是否覆盖最新位置。

## MQTT 约束

| Topic | 方向 | 用途 |
| --- | --- | --- |
| `devices/{device_id}/telemetry` | 设备到平台 | GPS、速度、方向、采集时间、上报时间。 |
| `devices/{device_id}/heartbeat` | 设备到平台 | 在线状态和最后活跃时间。 |
| `devices/{device_id}/box-events` | 设备到平台 | 开箱、关箱等安全事件。 |
| `drivers/{driver_id}/commands` | 平台到司机端 | 调度指令下发和撤销。 |
| `devices/{device_id}/commands` | 平台到设备 | 设备侧调度或控制指令。 |

Payload 必须包含 `event_id`、`occurred_at`、`reported_at` 和版本号。后端只接受服务端校验通过的消息进入最新状态计算。

## 地图与 ETA 约束

- ETA 计算通过 `EtaService` 封装，优先使用地图服务路线距离和耗时。
- 缺少当前位置、终点或供应商返回错误时，返回明确的不可计算状态，不返回伪造时间。
- 本地离线地图实现只用于开发和测试，使用直线距离与平均速度估算。
- 轨迹抽稀不得移除起点、终点、告警点、司机状态上报点和明显异常跳变点。

## RAG 与权限约束

- 知识库回答必须返回引用来源；无法确定时明确提示无法确认。
- 业务上下文检索必须先按角色过滤，再进入检索增强生成。
- 货主只能检索本人货物、运输任务、告警和问答历史；司机只能检索本人承运任务和指令。
- RAG 不作为权限裁决点；权限裁决在查询构造和 API 服务层完成。

## 本地启动命令契约

第一批工程骨架任务应落地以下命令：

```bash
docker compose -f infra/compose/dev.yml up --build
scripts/check.sh
```

`docker compose` 应启动 PostgreSQL、Redis、MQTT broker、API、worker 和 web。`scripts/check.sh` 是本仓库统一质量门禁；产品代码出现后必须逐步纳入：

- 后端格式、静态检查和单元测试。
- 前端格式、类型检查、单元测试和构建。
- 数据库迁移检查。
- 文档链接、冲突标记和秘密扫描。

## CI 约束

- 所有 PR 至少运行 `scripts/check.sh`。
- 引入产品代码后，CI 必须运行与本地相同的测试、类型检查和构建命令。
- CI 不依赖真实地图、真实 MQTT 云服务或真实 RAG 供应商；这些能力通过本地容器或测试替身验证契约。
